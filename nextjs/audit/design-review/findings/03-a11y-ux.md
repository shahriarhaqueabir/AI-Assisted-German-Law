# A11y & UX Flow Audit

## Summary

Audited 8 core files in the Next.js app for WCAG 2.2 AA compliance and UX flow friction. The app has a strong dark theme and visually polished interface, but accessibility fundamentals need attention: the language attribute is hardcoded to English despite 9 supported languages, focus indicators are broken by an undefined CSS variable, and critical form controls lack accessible names. Loading states, error announcements, and dropdown keyboard behavior also have gaps. 19 findings total — 5 critical, 7 high, 5 medium, 2 suggestions.

---

## Findings

### F-001: Hardcoded `lang="en"` ignores user's selected language
- **Severity**: critical
- **File**: `nextjs/src/app/layout.tsx` (line 21)
- **Current**: `<html lang="en" className={...}>` — hardcoded to English irrespective of the user's language selection in the language dropdown.
- **Issue**: WCAG 3.1.1 (Page Language). Screen readers use the `lang` attribute to select pronunciation rules. When a user switches to German (`de`), Turkish (`tr`), Arabic (`ar`), or any of the 9 supported languages, the page still announces content with English pronunciation. Search results, guidance text, and UI strings in the selected language will be mispronounced.
- **Recommendation**: Make `lang` dynamic based on the language setting stored in `ChatContext` / `localStorage`. Read the language from `useChat().settings.language` in the root layout and apply it to `<html lang={language}>`. For Arabic (`ar`), also set `dir="rtl"`.
- **Category**: a11y

### F-002: `:focus-visible` outline references undefined CSS variable — no focus indicator
- **Severity**: critical
- **File**: `nextjs/src/app/globals.css` (lines 199-202)
- **Current**:
  ```css
  :focus-visible {
    outline: 1px solid var(--accent-primary);
    outline-offset: 2px;
  }
  ```
  The variable `--accent-primary` is never defined in `:root` or anywhere else in the stylesheet. The CSS `var()` with no fallback for an undefined variable makes the entire `outline` declaration invalid.
- **Issue**: WCAG 2.4.7 (Focus Visible). Keyboard users navigating with Tab receive zero visible focus indication on any interactive element — links, buttons, inputs, selects. This affects the entire application.
- **Recommendation**: Replace with a defined color, e.g.:
  ```css
  :focus-visible {
    outline: 2px solid var(--accent-gold-bright, #c5a059);
    outline-offset: 2px;
  }
  ```
- **Category**: a11y

### F-003: Chat message input has no accessible label
- **Severity**: critical
- **File**: `nextjs/src/app/chat/page.tsx` (lines 605-616)
- **Current**: `<input type="text" value={input} ... placeholder="DESCRIBE SCENARIO..." ... />` with no `<label>`, no `aria-label`, and no `aria-labelledby`.
- **Issue**: WCAG 1.1.1 (Non-text Content) and 3.3.2 (Labels or Instructions). The placeholder alone is not a valid accessible label — it disappears on input and is not reliably exposed to assistive technology. Screen reader users will hear "edit text" or "blank" with no indication of the input's purpose.
- **Recommendation**: Add `aria-label="Describe your legal scenario"` or associate a `<label>` element using `htmlFor`/`id`.
- **Category**: a11y

### F-004: Chat send button has no accessible name
- **Severity**: critical
- **File**: `nextjs/src/app/chat/page.tsx` (lines 617-624)
- **Current**: `<button type="submit" disabled={...}>` contains only a `<Send className="w-5 h-5" />` icon. No text, no `aria-label`.
- **Issue**: WCAG 4.1.2 (Name, Role, Value). Screen readers announce "button" with no label, making it impossible to identify or activate the control by name.
- **Recommendation**: Add `aria-label="Send message"` to the button.
- **Category**: a11y

### F-005: Search input has no accessible label
- **Severity**: critical
- **File**: `nextjs/src/components/search-bar.tsx` (lines 38-46)
- **Current**: `<input type="text" value={query} ... placeholder="SEARCH STATUTE REPOSITORY..." ... />` with no `<label>`, no `aria-label`, and no `aria-labelledby`. The adjacent submit button (line 48) has `aria-label="Search"` but the input itself does not.
- **Issue**: WCAG 1.1.1 / 3.3.2 (same as F-003 but for the search input on every page containing this component — home page, search page).
- **Recommendation**: Add `aria-label="Search German laws"` to the `<input>`.
- **Category**: a11y

### F-006: Language and mode dropdowns do not close on Escape key
- **Severity**: high
- **File**: `nextjs/src/components/nav-bar.tsx` (lines 157-227, 229-288)
- **Current**: Both dropdowns use a fixed backdrop `<div>` to detect outside clicks (`onClick={() => setLangOpen(false)}`), but there is no `onKeyDown` handler for Escape. A keyboard user who opens the dropdown cannot close it without tabbing to a different element or activating another control.
- **Issue**: WCAG 2.1.2 (No Keyboard Trap). While this isn't a full trap (Tab still works), it creates a keyboard-only workflow failure: the dropdown persists and there is no standard escape mechanism.
- **Recommendation**: Add a `useEffect` that listens for `keydown` "Escape" on the window/document when the dropdown is open, or add an `onKeyDown` handler directly. Example:
  ```tsx
  useEffect(() => {
    if (!langOpen) return;
    const handler = (e: KeyboardEvent) => { if (e.key === 'Escape') setLangOpen(false); };
    document.addEventListener('keydown', handler);
    return () => document.removeEventListener('keydown', handler);
  }, [langOpen]);
  ```
- **Category**: a11y

### F-007: Loading states not announced to screen readers across all pages
- **Severity**: high
- **File**: Multiple — `nextjs/src/app/chat/page.tsx` (lines 575-593), `nextjs/src/app/search/page.tsx` (lines 49-56), `nextjs/src/app/auth/page.tsx` (lines 113-114), `nextjs/src/app/settings/page.tsx` (lines 738-750)
- **Current**: Loading indicators use visual animations (`animate-spin`, `animate-pulse`, `Loader2` icons) with text like "Retrieving Statutes..." or "Scanning Archives..." but none are wrapped in a `role="status"` or `aria-live="polite"` region.
- **Issue**: WCAP 4.1.3 (Status Messages). Screen readers will not announce loading state changes. Users relying on assistive technology cannot distinguish between a hung page and an active loading state.
- **Recommendation**: Wrap loading text/spinners in `<div role="status" aria-live="polite">` or use a VisuallyHidden component with `role="status"`. On the chat page's loading spinner (line 575-593), the same applies.
- **Category**: a11y

### F-008: Error and success messages lack `role="alert"`
- **Severity**: high
- **File**: `nextjs/src/app/auth/page.tsx` (lines 68-77), `nextjs/src/app/settings/page.tsx` (lines 617-621), `nextjs/src/app/search/page.tsx` (lines 57-60)
- **Current**: Error/success messages are rendered as plain `<div>` or `<p>` elements with no `role="alert"`:
  - Auth page: `<div className="bg-[#1a1a1a] border ...">{error}</div>` (line 68-72)
  - Settings key save: `<p className="text-[10px] text-accent-gold ...">{keyMessage}</p>` (line 617-621)
  - Search page: `<div className="p-8 bg-red-950/20 ...">{error}</div>` (line 57-60)
- **Issue**: WCAG 4.1.3 (Status Messages). Dynamic error/success content must be announced by screen readers without requiring focus. Without `role="alert"`, users may miss critical feedback (e.g., API key save failure, auth error, search failure).
- **Recommendation**: Add `role="alert"` to dynamic error/success message containers.
- **Category**: a11y

### F-009: Chat message log not exposed as a live region
- **Severity**: high
- **File**: `nextjs/src/app/chat/page.tsx` (lines 504-596)
- **Current**: The chat messages container is a plain `<div>` with `overflow-y-auto`. New messages are appended dynamically but the container has no `role="log"` or `aria-live="polite"`. Messages are also flat `<div>` elements, not list items.
- **Issue**: WCAG 4.1.3 (Status Messages). The expected pattern for chat logs is `role="log"` or `aria-live="polite"` so screen readers announce new messages as they arrive. Without this, users hear nothing when a response arrives.
- **Recommendation**: Add `role="log"` and `aria-live="polite"` to the scrolling message container. Consider wrapping messages in a `<ul>` with `<li>` items for proper list semantics.
- **Category**: a11y

### F-010: Color contrast failures on near-black backgrounds
- **Severity**: high
- **File**: Multiple — `nextjs/src/app/nav-bar.tsx` (line 146), `nextjs/src/app/settings/page.tsx` (lines 321-322, 266, 751), `nextjs/src/app/auth/page.tsx` (line 157)
- **Current**: Several text colors used on near-black backgrounds (glass panels, body `#0d0d0d`, elevated `#1a1a1a`) fail WCAG AA contrast thresholds:
  - `text-zinc-600` (#52525b) on `glass-panel-heavy` (effective ~#060606): ~2.7:1 — fail (needs 4.5:1).
  - `text-zinc-600` (#52525b) on `glass-panel` (effective ~#0b0b0b): ~2.6:1 — fail.
  - `text-[#6b6b6b]` on `bg-[#0d0d0d]` (auth page "← Back to search" link, line 157): ~3.6:1 — fail.
- **Issue**: WCAG 1.4.3 (Contrast Minimum). Text that does not meet 4.5:1 contrast is difficult or impossible to read for users with low vision.
- **Recommendation**: Use lighter text values on dark backgrounds. Replace `text-zinc-600` with `text-zinc-400` (#a1a1aa, ~7.9:1 on #060606) or `text-zinc-300` (#d4d4d8, ~14:1). For the `#6b6b6b` link, use `text-zinc-400` or `text-[#a3a3a3]` instead. Add a section to the tailwind config or design tokens with contrast-verified color pairs.
- **Category**: a11y

### F-011: Dropdown toggles missing `aria-expanded` and `aria-haspopup`
- **Severity**: high
- **File**: `nextjs/src/components/nav-bar.tsx` (lines 158-168, 230-237)
- **Current**: Both the language selector button (line 158) and mode selector button (line 230) have no ARIA state attributes. The language button has `title="Language / Sprache"` but neither has `aria-expanded` or `aria-haspopup="menu"`.
- **Issue**: WCAG 4.1.2 (Name, Role, Value). Screen reader users have no way to know that activating these buttons opens a menu, nor whether it is currently open or closed.
- **Recommendation**: Add `aria-haspopup="menu"` and `aria-expanded={langOpen}` (and `aria-expanded={open}` for mode) to the respective buttons.
- **Category**: a11y

### F-012: No skip-to-content link anywhere in the app
- **Severity**: high
- **File**: `nextjs/src/app/layout.tsx` (lines 20-34)
- **Current**: The root layout renders `<NavBar />` followed by `<div className="flex-1">{children}</div>`. There is no skip-to-content link before the navigation.
- **Issue**: WCAG 2.4.1 (Bypass Blocks). Keyboard and screen reader users must tab through all 4+ navigation items, the account link, language selector, and mode selector before reaching the main content on every page.
- **Recommendation**: Add a visually hidden skip link as the first focusable element inside `<body>`:
  ```tsx
  <a href="#main-content" className="sr-only focus:not-sr-only ...">
    Skip to content
  </a>
  ```
  Add `id="main-content"` to the main content container.
- **Category**: a11y

### F-013: Core navigation items hidden on mobile with no alternative
- **Severity**: high
- **File**: `nextjs/src/components/nav-bar.tsx` (lines 101-123)
- **Current**: The main nav links — Vault, Consult, Guidance, Archives — are wrapped in `<div className="hidden sm:flex sm:items-center sm:gap-2">`. Below `sm` breakpoint they are completely hidden. The only remaining navigation is the Vault logo (home) and the user/settings link.
- **Issue**: UX friction. On mobile, users cannot navigate to Chat, Guidance, or Bookmarks directly. They must either go through the home page or already know the URL. There is no hamburger menu or bottom nav for mobile.
- **Recommendation**: Add a mobile hamburger menu that reveals the nav links, or implement a bottom navigation bar for mobile. Use the existing sidebar pattern from the chat page as a model but make it available app-wide.
- **Category**: ux

### F-014: Mobile hamburger button in chat has `aria-label` but the sidebar lacks `aria-expanded` management
- **Severity**: medium
- **File**: `nextjs/src/app/chat/page.tsx` (lines 461-467, 448-453)
- **Current**: The mobile sidebar toggle has `aria-label="Toggle sidebar"` (good), but the sidebar's open state (`sidebarOpen`) is not communicated via `aria-expanded` on the toggle button. The `ConversationList` sidebar component receives `isOpen` and `onToggle` props but there's no focus management when it opens/closes.
- **Issue**: WCAG 4.1.2 (Name, Role, Value). Without `aria-expanded`, screen reader users don't know the sidebar state. Additionally, when the sidebar opens, focus is not moved into it; when it closes, focus is not returned to the toggle button.
- **Recommendation**: Add `aria-expanded={sidebarOpen}` to the toggle button. When the sidebar opens, move focus to its first focusable element; when it closes, return focus to the toggle button.
- **Category**: a11y

### F-015: Language/mode dropdowns do not trap focus when open
- **Severity**: medium
- **File**: `nextjs/src/components/nav-bar.tsx` (lines 170-226, 239-287)
- **Current**: When the language or mode dropdown is open, the backdrop div (fixed inset-0 z-10) blocks clicks but Tab focus can still move outside the dropdown menu to page content behind it.
- **Issue**: WCAG 2.4.3 (Focus Order) / UX friction. Users tabbing through the open dropdown can lose context by landing on elements behind the backdrop. While not a full keyboard trap violation, it creates a confusing experience — the dropdown stays open but focus is on elements obscured by the backdrop.
- **Recommendation**: When the dropdown is open, trap focus within the dropdown's menu element using a roving tabindex or a focus-trap utility. See the `focus-trap-react` library or implement a simple trap via `onKeyDown` and `useRef`.
- **Category**: a11y

### F-016: Mode selector buttons in settings lack `aria-pressed` or radio group semantics
- **Severity**: medium
- **File**: `nextjs/src/app/settings/page.tsx` (lines 224-282)
- **Current**: Four `<button>` elements act as mutually exclusive mode selectors (Local AI, Cloud AI, Browser AI, Basic Search). Active state is only communicated visually via CSS classes (`border-accent-gold/40`, `bg-accent-gold/10`).
- **Issue**: WCAG 4.1.2 (Name, Role, Value). Screen reader users hear "button" for each option with no indication that they are part of a single-choice group or which one is currently selected.
- **Recommendation**: Use `role="radio"` on each button with `aria-checked={isActive}`, wrapped in a container with `role="radiogroup"` and `aria-label="Chat mode"`. Alternatively, use semantic `<input type="radio">` elements styled as cards.
- **Category**: a11y

### F-017: No `role="navigation"` or `aria-label` on the nav element
- **Severity**: medium
- **File**: `nextjs/src/components/nav-bar.tsx` (line 90)
- **Current**: `<nav className="sticky top-6 z-50 w-full px-6">` — the `<nav>` element has no `aria-label`. With multiple `<nav>` or landmark regions on a page, screen reader users navigating by landmarks cannot distinguish this primary nav from other regions.
- **Issue**: WCAG 1.3.1 (Info and Relationships) / best practice. Multiple landmark regions should have distinct labels.
- **Recommendation**: Add `aria-label="Primary navigation"` to the `<nav>`.
- **Category**: a11y

### F-018: No `dir` attribute handling for Arabic (RTL) language
- **Severity**: medium
- **File**: `nextjs/src/app/layout.tsx` (line 21), `nextjs/src/app/globals.css`
- **Current**: Arabic (`ar`) is one of the 9 supported languages and the interface already has translated strings for it (see `useLanguage.ts`), but there is no logic to set `dir="rtl"` on `<html>` when Arabic is selected. The CSS has no RTL-aware rules.
- **Issue**: WCAG 1.3.2 (Meaningful Sequence) and 3.1.1. Arabic text is rendered left-to-right, breaking the reading order. Layout, alignment, and spacing that work for LTR scripts will be visually incorrect for Arabic users.
- **Recommendation**: Include `dir` in the dynamic `<html>` attribute alongside `lang`:
  ```tsx
  const dir = language === 'ar' ? 'rtl' : 'ltr';
  <html lang={language} dir={dir} ...>
  ```
- **Category**: a11y

### F-019: Auth "Initialize Session" link has empty `title` for unauthenticated users
- **Severity**: low
- **File**: `nextjs/src/components/nav-bar.tsx` (lines 127-144)
- **Current**: The user/settings link has `title={user?.email ?? ""}`. When `user` is null (unauthenticated), `title` becomes an empty string. The link text on mobile is just a `<User>` icon (no visible text), so the empty `title` means the link has no accessible name on small screens.
- **Issue**: WCAG 4.1.2 (Name, Role, Value). On mobile screens where the "Initialize Session" span is hidden (`hidden lg:inline`), the link has only a `<User>` icon with no `aria-label`. The empty `title` provides no accessible name.
- **Recommendation**: Add a conditional `aria-label`:
  ```tsx
  aria-label={user ? 'Settings' : 'Sign in'}
  ```
  Or always give it a meaningful `title` like `title={user?.email ?? "Sign in"}`.
- **Category**: a11y

### F-020: "Analyze via AI" button in search bar has no `aria-label`
- **Severity**: suggestion
- **File**: `nextjs/src/components/search-bar.tsx` (lines 55-64)
- **Current**: `<button type="button" onClick={...}>` with visible text "Analyze via AI" and a `<Brain>` icon. The text is visible, so screen readers will read it — this is acceptable for WCAG.
- **Issue**: Minor consistency issue. The search submit button on the same row has `aria-label="Search"` while this button doesn't need one (it has visible text). Not a violation, but the icon-only inner content uses icons without `aria-hidden`.
- **Recommendation**: Add `aria-hidden="true"` to the `<Brain>` icon inside the button so screen readers don't try to announce it as content. This is a minor polish fix.
- **Category**: suggestion

### F-021: Settings "Test Connection" result and "Saved" indicator lack `role="status"`
- **Severity**: medium
- **File**: `nextjs/src/app/settings/page.tsx` (lines 198-202, 738-750)
- **Current**: The "States Synchronized" badge (lines 198-202) and the connection test result span (lines 738-750) are rendered as plain `<span>` elements. They appear and disappear dynamically based on user action.
- **Issue**: WCAG 4.1.3 (Status Messages). Without `aria-live="polite"` or `role="status"`, screen reader users won't know when the save succeeded or when the connection test completed.
- **Recommendation**: Add `role="status"` and `aria-live="polite"` to both elements. For the saved badge, since it auto-hides after 2 seconds, ensure the live region content updates trigger announcement.
- **Category**: a11y
